Un cours de projet mené comme un vrai projet logiciel

Ce site présente une proposition de version personnalisée du cours INF 1430 — Projet en informatique (6 crédits, TELUQ), encadrée par Christian Jauvin.

Dans son format officiel, INF 1430 est un cours assez libre : l'étudiant réalise un projet informatique et remet quatre travaux écrits à intervalles réguliers. Ce format fonctionne, mais il laisse le professeur à distance du projet réel — il voit des rapports, pas le travail lui-même.

La version proposée ici renverse cette perspective : le projet est mené comme un vrai projet logiciel, avec les outils et les méthodes de l'industrie, et le professeur joue un rôle de chargé de projet et de reviewer plutôt que de simple correcteur. Les quatre travaux notés officiels sont conservés, mais ils deviennent des jalons naturels d'un processus continu et visible.

Les cinq piliers du modèle

  • 1. Le dépôt git

    Tout le projet vit dans un repo GitHub dont l'évolution est visible en continu : commits, branches, tags. Pas de « boîte noire » entre deux remises.

  • 2. Les pull requests

    Le travail avance par PR, que le professeur review : commentaires, demandes de changement, approbation. La rétroaction est continue et concrète.

  • 3. GitHub Projects

    Planification et suivi via un board : backlog, itérations, issues, milestones. Une gestion inspirée de scrum, adaptée à un projet individuel.

  • 4. La communication

    Asynchrone par défaut (issues, PRs, discussions GitHub — tout laisse une trace), synchrone quand c'est utile (rencontres Teams).

  • 5. L'alignement avec INF 1410

    Les méthodes pratiquées ici sont l'application directe des idées de génie logiciel du nouveau cours INF 1410. INF 1430 en devient le laboratoire.

Compatible avec le format officiel

Les quatre travaux notés du cours officiel restent les jalons d'évaluation, avec les mêmes pondérations (10 %, 20 %, 30 %, 40 %). Ce qui change, c'est la manière d'y arriver : chaque travail devient le point culminant d'un processus documenté dans le repo, plutôt qu'un document produit en vase clos. Le détail se trouve dans la page Déroulement.

Pour commencer

  • Le modèle — le fonctionnement concret : repo, PRs, board, communication.

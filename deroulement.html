Le déroulement du semestre

Le cours conserve les quatre travaux notés du format officiel, avec leurs pondérations. Le modèle les réinterprète comme les jalons d'un processus continu, visible dans le repo du début à la fin.

Les quatre travaux notés (format officiel)

# Titre Pondération
1Plan de projet documenté10 %
2Rapport mi-parcours20 %
3Produit final fonctionnel30 %
4Rapport du projet présenté40 %

Leur réinterprétation dans le modèle

Travail officiel Dans le modèle proposé
1. Plan de projet (10 %) La mise en place du projet : repo créé, board GitHub Projects configuré, backlog initial d'issues, plan de projet documenté dans le repo (README ou document dédié). Le plan n'est pas un document théorique : c'est l'état initial, réel et fonctionnel, de l'infrastructure du projet.
2. Rapport mi-parcours (20 %) Un milestone à mi-parcours : état du board, PRs mergées jusque-là, et rapport intermédiaire versionné dans le repo. Le rapport s'appuie sur la trace accumulée (commits, PRs, issues) — il synthétise un avancement déjà constatable.
3. Produit final (30 %) Le repo lui-même : le produit fonctionnel, mais aussi son historique de développement, la qualité du code et celle des PRs qui y ont mené. Le « dépôt » du produit final est un tag/release dans le repo, pas un fichier zip envoyé par courriel.
4. Rapport final (40 %) Le rapport final, rédigé (et idéalement versionné) dans le repo, incluant un retour réflexif sur le processus : ce qui a bien fonctionné, ce qui a été ajusté en cours de route, ce que la méthode a apporté au projet.

Le semestre, par phases

À titre indicatif, pour un semestre de 15 semaines (le découpage exact est convenu avec chaque étudiant selon son projet) :

Phase Semaines Contenu
Démarrage 1 – 2 Définition du projet avec le professeur ; création du repo et du board ; backlog initial ; rédaction du plan de projet. → Travail 1.
Itérations, première moitié 3 – 7 Développement par itérations : issues → branches → PRs → reviews → merges. Ajustements du backlog au fil des apprentissages.
Mi-parcours 8 Milestone de mi-parcours : bilan de l'avancement, rapport intermédiaire, rencontre Teams de fond. → Travail 2.
Itérations, seconde moitié 9 – 13 Poursuite du développement, avec un accent croissant sur la complétude, la robustesse et la documentation du produit.
Finalisation 14 – 15 Gel des fonctionnalités ; release finale du produit (→ Travail 3) ; rédaction du rapport final avec retour réflexif (→ Travail 4).

La place exacte du processus (qualité des PRs, activité sur le board, communication) dans la note de chaque travail est un point en cours de précision ; l'esprit du modèle est que un processus sain rend presque mécaniquement les livrables meilleurs.

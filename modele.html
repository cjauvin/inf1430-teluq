Le modèle, concrètement

Quatre mécanismes, tous appuyés sur GitHub : le dépôt, les pull requests, le board de gestion, et la communication.

Le dépôt GitHub

Dès le début du cours, l'étudiant crée un dépôt GitHub qui devient la colonne vertébrale du projet : tout y vit — le code, bien sûr, mais aussi la documentation, les rapports, et les décisions (via les issues et les PRs). Le professeur y a un accès permanent en tant que collaborateur.

  • Le dépôt peut être privé (avec le professeur comme collaborateur) ou public, au choix de l'étudiant.
  • L'historique de commits doit refléter le travail réel : des commits réguliers, avec des messages descriptifs, plutôt qu'un gros dépôt de code la veille d'une remise.
  • Le README du dépôt sert de porte d'entrée : description du projet, instructions d'installation et d'exécution, état d'avancement.

Le flux de travail par pull requests

Le travail avance par unités cohérentes — une fonctionnalité, une correction, un morceau de documentation — chacune développée sur une branche et proposée par une pull request :

  1. Brancher : l'étudiant crée une branche pour la tâche en cours (idéalement liée à une issue du board).
  2. Ouvrir une PR : quand le travail est prêt (ou assez avancé pour mériter un regard), l'étudiant ouvre une pull request qui décrit le changement.
  3. Review du professeur : commentaires, questions, suggestions, demandes de changement — directement sur le code, ligne par ligne au besoin.
  4. Itérer : l'étudiant répond, corrige, pousse de nouveaux commits sur la même PR.
  5. Merger : une fois la PR approuvée, elle est mergée dans main.

C'est le cœur pédagogique du modèle : chaque PR est une occasion de rétroaction concrète et située. La qualité attendue des PRs (taille raisonnable, description claire, réponse aux commentaires) fait partie des attentes du cours.

GitHub Projects : la gestion du projet

La planification et le suivi passent par un board GitHub Projects attaché au dépôt, selon une méthode inspirée de scrum et adaptée à un projet individuel :

  • Backlog : les tâches du projet, exprimées en issues, priorisées et raffinées au fil du semestre.
  • Itérations : le semestre est découpé en itérations courtes (cadence à convenir, p. ex. deux semaines) ; au début de chacune, l'étudiant choisit les issues qu'il compte réaliser.
  • Milestones : les quatre travaux notés correspondent à des milestones du projet, ce qui relie la gestion quotidienne aux jalons officiels du cours.
  • Board : les colonnes (p. ex. À faire / En cours / Terminé) donnent en un coup d'œil l'état du projet.

La communication

Deux canaux complémentaires, avec une règle simple :

Asynchrone d'abord (GitHub)

La voie par défaut : commentaires d'issues et de PRs, discussions du dépôt. Ce mode laisse une trace consultable, permet de réfléchir avant de répondre, et n'exige pas de coordonner des horaires. C'est aussi le mode de travail normal des équipes logicielles distribuées.

Synchrone quand c'est utile (Teams)

Des rencontres Teams en temps réel, planifiées au besoin : pour débloquer une situation qui piétine à l'écrit, discuter d'une orientation importante, ou simplement faire le point. La fréquence exacte (fixe ou sur demande) est convenue en début de cours.

Dans les deux cas, l'attente centrale est une communication active : le professeur s'engage à une bonne réactivité sur les PRs et les questions ; en retour, l'étudiant maintient un rythme de travail visible et signale tôt les difficultés — c'est précisément ce que le modèle rend possible.

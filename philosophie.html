Philosophie

Quatre idées de fond structurent ce modèle. Aucune n'est nouvelle en soi — ce sont des pratiques établies du développement logiciel professionnel. La proposition consiste à en faire la matière même du cours.

Apprendre le génie logiciel en le pratiquant

Un cours de projet est l'occasion idéale de pratiquer le génie logiciel pour vrai : non pas résoudre des exercices sur la gestion de versions ou la revue de code, mais gérer réellement ses versions et faire réellement reviewer son code, sur un projet qui compte. Les outils choisis (git, GitHub, les pull requests, un board de gestion) ne sont pas des contraintes administratives : ce sont les outils réels de l'industrie, et savoir les utiliser avec aisance est en soi un des apprentissages les plus durables du cours.

La rétroaction continue plutôt que la correction terminale

Dans le format classique, le professeur découvre le travail au moment des remises : si une mauvaise décision a été prise en semaine 3, elle est découverte — et pénalisée — en semaine 8. Dans le modèle proposé, le professeur voit le travail au fur et à mesure, par les pull requests et le board. La rétroaction arrive quand elle est encore utile, c'est-à-dire quand il est encore temps d'ajuster le tir. La review de code devient le principal mécanisme d'encadrement pédagogique : concrète, ancrée dans le code de l'étudiant, et bidirectionnelle (l'étudiant répond, argumente, corrige).

Ce déplacement change aussi la nature de l'évaluation : les travaux notés ne sont plus des moments de vérité stressants, mais des jalons qui consolident un travail déjà vu, déjà commenté, déjà amélioré.

La trace écrite comme mémoire du projet

Tout ce qui passe par git et GitHub laisse une trace : l'historique des commits raconte le développement, les conversations de PR documentent les décisions, les issues et le board racontent la planification et ses ajustements. Cette trace a une triple valeur :

  • pour l'étudiant, c'est la mémoire du projet — et la matière première des rapports à rédiger, qui deviennent en bonne partie une synthèse de choses déjà écrites ;
  • pour le professeur, c'est une visibilité honnête sur le processus réel, pas seulement sur le produit final ;
  • pour l'évaluation, c'est une base factuelle et constatable, plutôt qu'une reconstruction après coup.

C'est pourquoi la communication asynchrone, dans GitHub, est la voie par défaut : elle laisse une trace, elle laisse le temps de réfléchir, et elle correspond au mode de travail réel des équipes distribuées. Les rencontres synchrones (Teams) restent disponibles pour ce que l'écrit fait mal : débloquer une situation, discuter d'une orientation, faire le point.

L'alignement avec INF 1410 (Génie logiciel)

Ce modèle vient directement des idées exposées dans la nouvelle version du cours INF 1410 — Génie logiciel (prévue pour septembre 2026). Là où INF 1410 présente les concepts — gestion de versions, revue de code, intégration continue des changements, planification itérative, communication d'équipe — INF 1430 les fait pratiquer sur un projet réel, de bout en bout.

INF 1430 devient en quelque sorte le laboratoire d'INF 1410 : les méthodes qu'on y pratique sont l'application concrète des idées de génie logiciel qu'INF 1410 présente.

Des renvois explicites vers le contenu d'INF 1410 seront ajoutés ici lorsque ce cours sera offert.
